### Problem: cu121 Incompatible with RTX 5070 Ti

The March 30 session note specified reinstalling PyTorch with `cu121`. On execution, the
GPU was detected but produced the following warning:


support:

```bash
    torch torchvision \
    --index-url https://download.pytorch.org/whl/cu128 \
    --force-reinstall
```

**Verified output:**

## 5. Conceptual Explanations Generated

The following explanations were generated this session for team communication. These are
preserved here as authoritative references.

### 5.1 What Training, Validation, and Testing Are

**Training (70% of data):** The model learns from this data. Every weight update happens
only here. Images are fed in batches of 64. The model makes predictions, loss is computed,
gradients are backpropagated, and weights are updated by AdamW. This happens continuously
in a loop — one full pass through all training images is one epoch.

**Validation (15% of data):** Shown to the model during training but never used for
weight updates. Think of it as a practice exam the model can see its score on but cannot
study from directly. After every epoch, the model runs inference on val — no gradients,
no updates. Macro F1 is computed. If it improves, the checkpoint is saved. If it doesn't
improve for 5 consecutive epochs, early stopping triggers. Validation is what drives
every checkpoint and stopping decision.

**Test (15% of data):** The real exam. Completely hidden until training finishes. Touched
exactly once per run, after training, using the best checkpoint selected by validation F1.
Produces the final honest performance numbers: accuracy, macro F1, per-class
precision/recall/F1, and confusion matrix. Test is never used during training for any
reason — doing so would leak information and make the numbers optimistic.

### 5.2 What Weights Are

A weight (also called a parameter) is a floating-point number stored inside the model.
ResNet50 has 24.6 million of them. These numbers are what training actually changes.

The model is a mathematical function: input is a 224×224 image (roughly 150,000 pixel
values), output is 4 numbers (one per class). The weights are the internal dials that
define exactly how that function behaves. At the start of fine-tuning they are set to
the ImageNet-pretrained values. After each batch, they get nudged slightly in the
direction that would have produced a more correct answer. By the end of training, 24.6
million numbers have been precisely adjusted so that feeding in a diamond image produces
a high score for the correct tier and low scores for the other three.

### 5.3 What Logits, Softmax, and Cross-Entropy Loss Are

The model's raw output for a single image is a vector of 4 numbers called **logits** —
one per class. These are unnormalized scores. Higher logit means the model is more
confident that class is correct.

**Softmax** converts logits to probabilities by exponentiating each and dividing by the
sum, forcing all four to add up to 1.0:

```
budget:           0.18   (18% confident)
mid_range:        0.68   (68% confident)  ← predicted class
premium:          0.09
investment_grade: 0.04
```

**Cross-entropy loss** then measures how wrong the prediction was. If the true label is
mid_range and the model gave it 68% probability, the loss is low. If it gave mid_range
3% probability, the loss is very high. Loss is a single number summarizing total wrongness
across the entire batch. Backpropagation uses this number to compute which direction to
nudge every weight.

### 5.4 What Macro F1 Is and Why It's the Primary Metric

**Accuracy** is the fraction of all predictions that were correct. Easy to read but
misleading with class imbalance — a model that predicts mid_range for everything would
get ~48% accuracy on ja_natural without learning anything useful.

**Macro F1** computes F1 separately for each of the 4 classes and takes the unweighted
average — each class counts equally regardless of size. F1 per class is the harmonic
mean of precision and recall:

- **Precision:** Of all diamonds predicted as investment_grade, what fraction actually were?
- **Recall:** Of all actual investment_grade diamonds, what fraction did the model find?
- **F1:** `2 × (precision × recall) / (precision + recall)`

Macro F1 correctly penalizes the model for failing on investment_grade (10.8% of data)
just as much as failing on mid_range (48% of data). This is why macro F1 — not accuracy
— drives checkpoint selection and all reporting.

### 5.5 What Backpropagation and Gradients Are

The **gradient** of the loss with respect to a weight is a derivative — it tells you
which direction to nudge that weight to reduce the loss, and by how much. With 24.6
million weights, backpropagation computes 24.6 million partial derivatives simultaneously
using the chain rule propagated backwards through every layer of the network.

**AdamW** then applies those gradients. It maintains a running estimate of how large
each weight's gradients have been historically and scales updates accordingly. Weights
with historically large gradients get smaller updates; weights with small gradients get
proportionally larger updates. Weight decay (1e-4) gently shrinks all weights toward zero
each step — preventing any single weight from growing so large the model overfits.

### 5.6 What Differential Learning Rates Are

The backbone (pretrained layers) gets LR = 3e-5. The head (randomly initialized) gets
LR = 3e-4 — ten times larger. The backbone already has good weights from ImageNet
pretraining. Large updates would destroy that. The head starts from scratch and needs
bigger steps. Same optimizer, same scheduler, different step sizes.

### 5.7 What the Cosine Annealing Schedule Is

The learning rate follows the shape of a cosine curve — starting at the full rate (3e-4
for head, 3e-5 for backbone) and smoothly decreasing to a minimum (1e-6) by epoch 30.
Early in training you want large steps to explore the loss landscape broadly. Later you
want small precise steps to settle into the best minimum without overshooting. Cosine
annealing is smoother than step decay (sudden drops) and tends to produce better final
performance.

### 5.8 What AMP and GradScaler Are

AMP (Automatic Mixed Precision) runs forward and backward passes in float16 instead of
float32. float16 uses half the memory and roughly doubles throughput on modern GPUs which
have dedicated float16 hardware. The RTX 5070 Ti has extensive float16 (Tensor Core)
support.

The risk of float16 is that very small gradient values can underflow to zero (float16
has reduced numerical range). GradScaler compensates by scaling loss values up before
the backward pass and scaling gradients back down before the weight update, keeping
values in a range float16 can represent accurately.

### 5.9 What False Positives and False Negatives Are

These come from the confusion matrix — a 4×4 grid where cell [i][j] means "true class
was i, model predicted j." The diagonal is correct predictions. Off-diagonal cells are
mistakes.

- **True positives:** Diagonal cells — predicted correctly
- **False positives for class X:** Column X, off-diagonal — predicted as X when it wasn't
- **False negatives for class X:** Row X, off-diagonal — was X but predicted as something else

For example: cell [premium][mid_range] is "true class was premium, model predicted
mid_range." High precision but low recall on investment_grade means: when the model says
investment_grade it's usually right (low false positives), but it misses more than half
the actual investment_grade diamonds (high false negatives), pushing them into mid_range
or premium.

The classification_report.txt saved per run shows precision, recall, F1, and support
per class. The confusion_matrix in final_metrics.json shows the full 4×4 grid.

### 5.10 Why We Did the RF Baseline

The RF baseline establishes the **performance ceiling** — the best any model could
reasonably achieve given near-perfect information. Without it, an image model accuracy
of 67% has no frame of reference.

The RF receives the 4C attributes directly (carat, cut, color, clarity) — the same
attributes professional gemologists use to price diamonds, and the same attributes used
to compute the tier labels. It effectively reverse-engineers the pricing formula we used
to create the labels. It does not see any images.

The ceiling interpretation: carat dominance (0.79–0.85 importance) confirms that diamond
price is legitimately driven by carat, not that the labels are trivial. The remaining
15–21% from cut/color/clarity confirms the labels reflect real multi-attribute pricing.
The RF cannot reach 100% because price also reflects market conditions, availability,
and retailer markup that the 4Cs don't fully capture.

The image-to-tabular gap (0.18–0.27 F1) quantifies the cost of working from retail
photographs. A JPEG cannot communicate carat weight with the precision of a scale
measurement. This is the paper's honest answer to Research Question 3: no visual
architecture approaches the tabular ceiling, and the gap is physically explained.

### 5.11 Classification vs Regression

This project uses **classification** — the model predicts which of four categories a
diamond belongs to (budget, mid_range, premium, investment_grade). Output is 4 logits,
one per class.

**Linear regression** would predict a continuous real number — the actual price in
dollars. It would require replacing the 4-output head with a 1-output head and swapping
cross-entropy loss for mean squared error or Huber loss. Log-transforming price first
would be necessary because diamond prices are right-skewed.

A regression model would sidestep the tier boundary approximation problem entirely —
no need to define where budget ends and mid_range begins. It was discussed as a possible
paper extension. Conclusion: worth raising with the professor, but outside committed
scope. Would require ~2–3 hours to implement as a separate script.

### 5.12 How the Two-Stage Pipeline Works at Inference

The same single image flows through both stages:

1. Input: retail diamond photograph
2. Stage 1 classifies gemstone type (diamond vs ruby vs emerald vs sapphire, etc.)
3. If NOT diamond → rejected with gemstone type label
4. If diamond → same image passed to Stage 2
5. Stage 2 classifies into budget / mid_range / premium / investment_grade
6. Output: value tier prediction

JA and BE images are diamonds only and feed Stage 2 training exclusively. Stage 1 training
uses Sindhu + Kamath + Purswani gemstone datasets. Mixing them would bias Stage 1 toward
predicting diamond for everything.

End-to-end accuracy = Stage 1 diamond recall × Stage 2 accuracy. If Stage 1 finds
diamonds 95% of the time and Stage 2 gets the tier right 67% of the time, end-to-end
accuracy is ~63%.
