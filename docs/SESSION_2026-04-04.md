## 2. Cross-Domain Training — Challenge Log

### Challenge 1 — Corrupted Image Crashed DataLoader at Epoch 29

**Run affected:** resnet50 / ja_natural / cross

The first cross-domain run trained cleanly for 28 epochs, saving a best checkpoint at
val F1=0.6707. At epoch 29 DataLoader worker 7 crashed mid-batch:

```
OSError: Caught OSError in DataLoader worker process 7.
OSError: broken data stream when reading image file
```

Simultaneously, train loss became NaN and val F1 collapsed to 0.1046 — the NaN
propagated through all 24.6M weights in a single backward pass. The run terminated
with an unhandled exception.

**Root cause:** `investment_grade/8093578.jpg` — a JPEG with a readable header but
corrupted data stream. The March 30 image audit checked file size and resolution but
did not perform a full PIL decode, so the corruption was invisible at audit time.

**Diagnosis:** Full PIL decode scan of all 86,149 images in the cross-train split:

```python
from PIL import Image
import pandas as pd

csv = "data/splits/ja_natural_cross_train.csv"
image_dir = "ja_scraper/output/images"
df = pd.read_csv(csv)
bad = []
for i, row in df.iterrows():
    path = f"{image_dir}/{row['value_tier']}/{row['diamond_id']}.jpg"
    try:
        img = Image.open(path)
        img.load()
    except Exception as e:
        bad.append((path, str(e)))
```

Result: exactly 1 corrupted file confirmed.

**Resolution:**
1. Deleted corrupted file: `rm ja_scraper/output/images/investment_grade/8093578.jpg`
2. Patched `src/diamond_dataset.py` `__getitem__` to handle corrupt files gracefully:

```python
# Before
image = Image.open(row["_img_path"]).convert("RGB")

# After
try:
    image = Image.open(row["_img_path"]).convert("RGB")
except Exception:
    image = Image.new("RGB", (224, 224), (128, 128, 128))
```

This returns a neutral gray 224×224 image for any corrupt file rather than crashing
the DataLoader. The fix is permanent and applies to all future runs across all splits
and datasets.

### Challenge 2 — Resume Flag Scrambled Weights (Learning Rate Mismatch)

**Run affected:** resnet50 / ja_natural / cross (attempt 2)

With the checkpoint from epoch 28 saved and the corrupted image deleted, the natural
recovery was `--resume`. The weights loaded correctly but the cosine annealing scheduler
reset to epoch 1 — jumping the learning rate back to 3e-4 (full starting rate) instead
of continuing from near the cosine minimum (~1e-6 at epoch 28).

Training ran for 11 more epochs with a massively inflated learning rate on an
already-converged model. JA val F1 looked plausible (0.67) because val distribution
matched training. The actual weights were silently destroyed.

When the best checkpoint was evaluated on BE test: **macro F1=0.0765**. The model
predicted "premium" for 93% of all BE images — complete collapse.

```
TEST RESULTS [resnet50__ja_natural__cross — bad resume]
  Accuracy : 0.1119
  Macro F1 : 0.0765
  budget F1: 0.0024  investment_grade F1: 0.0286
  mid_range F1: 0.0843  premium F1: 0.1906
  premium recall: 0.93  ← predicting premium for everything
```

**Root cause:** `--resume` restores model weights but not optimizer state or scheduler
state. The scheduler has no memory of having already completed 28 epochs.

**Resolution:** Deleted checkpoint, retrained from scratch with no `--resume`. The
`--resume` flag as implemented is only safe for early-epoch recovery (e.g. epoch 3 of
30 after a power outage). Using it after a near-complete run produces an LR mismatch
that silently destroys the model without any error or warning.

**Logged as known limitation of train.py:** A proper resume implementation requires
saving and restoring `scheduler.state_dict()` and `optimizer.state_dict()` alongside
`model.state_dict()`. Not implemented this session — out of scope for paper deadline.

### Challenge 3 — AMP NaN Instability on Cross-Domain Data Distribution

**Run affected:** resnet50 / ja_natural / cross (attempts 3 and 4)

With the corrupted image deleted and no `--resume`, the run hit NaN loss again at
epoch 14 — a different epoch each time, same collapse pattern.

```
Epoch 14: nan  0.7917  0.7748    nan  0.2646  0.1046
Epoch 15: 0.5522  0.8021  0.7853    nan  0.2646  0.1046
Epoch 16: 0.5390  0.8099  0.7940    nan  0.2646  0.1046
```

Val loss became NaN permanently from epoch 14 onward. Train loss recovered (the
forward pass on new batches was fine) but the network weights were already corrupted.

**Root cause:** AMP (Automatic Mixed Precision) numerical instability. Float16 can
represent values up to ~65,504. The GradScaler scales gradients to prevent float16
underflow, but if a gradient spike is extreme enough, even after the GradScaler's
correction the values can still overflow float16 to infinity — which propagates as
NaN through the backward pass.

**Why cross-domain and not within-site:** The cross-domain training split has a
different composition than the within-site split. The WeightedRandomSampler constructs
batches from a different distribution that, at some point around epoch 13–14, produced
a gradient spike exceeding what float16 could safely represent. Within-site runs
never encountered this region of the loss landscape across all 12 runs.

**Resolution:** Disabled AMP entirely. Removed GradScaler, removed all `autocast`
contexts, reverted training branch to clean float32:

```python
# Removed from train.py:
scaler = GradScaler("cuda")

# Training branch before:
with autocast("cuda"):
    logits = model(images)
    loss = criterion(logits, labels)
scaler.scale(loss).backward()
scaler.unscale_(optimizer)
nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
scaler.step(optimizer)
scaler.update()

# Training branch after:
logits = model(images)
loss = criterion(logits, labels)
loss.backward()
nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
optimizer.step()
```

Float32 can represent values 10³⁴× larger than float16. The same gradient spike that
breaks float16 is completely safe in float32. Tradeoff: ~15–20% slower epochs
(~175s vs ~96s) due to lower float32 throughput vs float16 Tensor Cores.

**Academic disclosure required:** Cross-domain runs used float32 (AMP disabled for
numerical stability) vs within-site runs which used AMP (float16). This is a
methodological difference that must be disclosed in the paper:

> Cross-domain experiments were conducted in float32 precision following observed
> AMP numerical instability on the cross-domain data distribution. Within-site
> experiments used AMP (float16 + GradScaler). All other hyperparameters were
> identical between experiment types.

### Challenge 4 — Early Stopping Miscalibrated for Cross-Domain Dynamics

**Run affected:** resnet50 / ja_natural / cross (attempt 4, float32, clean)

With AMP disabled, the run completed without NaN or crash. But early stopping fired
at epoch 8 with val F1=0.5991 as best:

```
Epoch 3: val F1=0.5991  ← best
Epoch 4: 0.5752
Epoch 5: 0.5814
Epoch 6: 0.5815
Epoch 7: 0.5858
Epoch 8: 0.5921  ← still improving but patience expired at 5 from epoch 3
```

The model was still climbing (0.5752 → 0.5921) but patience=5 from the previous
best killed it before it could improve further. Within-site runs improved steadily
for 25–30 epochs. Cross-domain val curves oscillate more — the model is learning
from one distribution (JA) while validated on a held-out portion of that same
distribution, producing noisier epoch-to-epoch variation.

**Resolution:** Increased patience to 10 and epochs to 50 for all cross-domain runs:

```bash
sed -i 's/PATIENCE=5/PATIENCE=10/' scripts/run_cross_domain.sh
sed -i 's/EPOCHS=30/EPOCHS=50/' scripts/run_cross_domain.sh
```

**Academic disclosure required:**

> Within-site experiments used epochs=30, patience=5. Cross-domain experiments used
> epochs=50, patience=10, motivated by greater val F1 oscillation observed in
> cross-domain training dynamics. The hyperparameter change was motivated by an
> observed training dynamic, not post-hoc result optimization.

## 3. JA→BE Transfer Failure — Confirmed Finding

After all four fixes, the full 47-epoch run of resnet50/ja_natural/cross completed
cleanly. The BE test result:

```
TEST RESULTS [resnet50__ja_natural__cross — 47 epochs, patience=10, float32]
  Accuracy : 0.1701
  Macro F1 : 0.1458
  budget F1: 0.0144  investment_grade F1: 0.2129
  mid_range F1: 0.1644  premium F1: 0.1918
```

| Metric | Value |
|---|---|
| JA val F1 (best, epoch 37) | 0.6794 |
| BE test F1 | 0.1458 |
| Cross-domain accuracy delta | −0.534 F1 points |

The model learned JA well (val F1=0.679) but failed catastrophically on BE (test
F1=0.146). Consistent across all four attempts at different stopping points (epoch
8, 12, 28, 47) — all producing BE test F1 of 0.08–0.15.

**Interpretation:** ResNet50 trained on JA's 40× superzoom images (757×600, controlled
white background, standardized lighting) learned site-specific visual artifacts that
do not transfer to BE's 300×300 grey-background multi-supplier thumbnails. The two
imaging domains are sufficiently different that cross-site visual features are not
shared at the ResNet50 backbone level.

**This is a paper finding, not a training failure.** The result directly answers
Research Question 4: JA→BE transfer fails for ResNet50 on natural diamonds, with a
cross-domain accuracy delta of −0.53 F1 points. Whether this generalizes across
architectures and the ja_lab subset remains to be seen from the remaining 11 runs.

**The bad result was preserved:**

```bash

      results/training/resnet50__ja_natural__cross__attempt1_bad
```

The attempt1_bad directory documents the methodology challenges for the paper's

## 4. System Instability — protouno After 48 Hours

After approximately 48 continuous hours of training, protouno exhibited:
- Slow tab opening in browser
- Complete failure to load web pages
- apt package list corruption (partial write during memory pressure)
- Network connection instability

**Diagnosis:** Memory exhaustion under zero-swap configuration.

- Training process: ~3.4GB RAM + 3.4GB VRAM continuously held
- 8 DataLoader workers: ~1.9GB RAM each = ~15.2GB
- Total training footprint: ~18–19GB of 31GB available
- With no swap (confirmed: `MiB Swap: 0.0 total`) the kernel cannot page out
  under pressure — the system slows to a crawl rather than swapping
- Over 48 hours, memory fragmentation builds until new allocations (browser tabs,
  DNS resolution buffers) cannot be satisfied

**apt corruption:** apt package lists in `/var/lib/apt/lists/` received a partial
write during peak memory pressure. Resolution: deleted the corrupted list file,
re-ran `apt update`. List files are always safe to delete and re-fetch.

**Resolution:** Full reboot. Cleared memory fragmentation, restarted all daemons,
renewed DHCP lease, restored network connectivity.

**Mitigation for future long runs:** Consider reducing `NUM_WORKERS` from 8 to 6
during cross-domain runs (larger datasets = more RAM per worker) to reduce sustained
memory pressure. GPU utilization impact will be minimal.

## 8. Key Decisions Logged

**Why the bad JA→BE result was preserved and not discarded:**
The F1=0.1458 result for resnet50/ja_natural/cross is itself a finding. It documents
that JA→BE transfer fails consistently across multiple attempts, hyperparameter
settings, and training durations. It belongs in the paper's cross-domain results
table with a clear note that the result reflects genuine domain gap failure, not
a training artifact. Archived to `resnet50__ja_natural__cross__attempt1_bad/`.

**Why the full 12-run suite is being run rather than stopping after one JA→BE failure:**
One architecture on one subset (resnet50/ja_natural) is insufficient to conclude
that all JA→BE transfer fails. EfficientNetV2 and ViT may behave differently.
The ja_lab subset may transfer differently from ja_natural. The full suite is
required to make the cross-domain claim cleanly in the paper.

**Why BE→JA is the higher-interest experiment:**
BE's varied multi-supplier imaging may produce more generalizable features than
JA's controlled 40× regime. A model trained on diverse BE images might transfer
to JA's consistent images more successfully than the reverse. If BE→JA succeeds
while JA→BE fails, the finding is asymmetric and architecturally interesting:
diversity in training imaging aids cross-domain generalization; consistency hurts it.

## 9. End-to-End Pipeline Evaluation — Design Discussion

A gap was identified: no existing dataset has both gemstone type labels and diamond
value tier labels simultaneously. The three options discussed:

**Option 1 — Construct from existing data (recommended):**
Combine MDPI non-diamond images + stratified sample from JA/BE test splits (already
labeled with tiers, never seen during training). ~1,500 images total. Provides real
Stage 1 gating decisions and measurable Stage 2 accuracy. Requires ~2–3 hours to
build. Must be raised with professor before implementation.

**Option 2 — Theoretical calculation:**
`End-to-end accuracy = Stage 1 diamond recall × Stage 2 accuracy`
Valid but weaker than empirical measurement.

**Option 3 — Disclose as limitation:**
Note that a proper end-to-end evaluation requires a dataset with both gemstone type
and value tier labels — which does not exist publicly. Frame as future work.

The end-to-end test set should include images from both JA and BE to avoid evaluating
Stage 2 only on its own imaging domain. Testing only on JA diamonds after JA training
is easy mode. A rigorous pipeline test requires both sites.

**Decision:** Raise with professor before building. Not yet implemented.
